### 2.3 API-Key Masking & Source-Corruption

Im `read_file`-Output mehrerer Dateien wird `apiKey: ***` angezeigt. Das ist hier nur ein Display-Problem der Hermes-Ausgabe, aber **beim Patching in der Nähe von `apiKey` muss unbedingt mit `xxd`/`grep`/`npm run build` verifiziert werden**, dass die Source nicht wirklich korrumpiert wird (siehe Skill-Regel 11). Aktuell im Code kein Fehler, aber ein wiederkehrendes Risiko.

### 2.4 Code-Qualität & potentielle Bugs

#### a) `AppController` — `sendMessage` Bridge startet Agent-Run ohne try/catch

In `AppController.handleBridgeRequest` (case `sendMessage`):

```ts
this.agent.run(req.text, config, this.currentSessionId || undefined).catch((e) => {
  chat?.appendError(e instanceof Error ? e.message : String(e));
  chat?.setStatus('idle');
  this.isRunning = false;
});
```

- ✅ Error-Handling ist vorhanden (`.catch`).
- ⚠️ `this.isRunning` wird auf `true` gesetzt, bevor geprüft wird, ob `agent.run` tatsächlich losläuft. Wenn `run` synchron throwed (passiert im aktuellen Code nicht, aber bei zukünftigen Änderungen möglich), bleibt `isRunning = true` hängen.
- ⚠️ `this.agent.run` wird ohne `try/catch` vor `this.isRunning = true` aufgerufen. Empfohlen: `try { ... await ... } finally { this.isRunning = false; }`.

**Empfehlung:** `sendMessage` analog zum `onSubmit` in `registerApps` mit `try/await/finally` umbauen, damit `isRunning` immer zurückgesetzt wird.

#### b) `WindowManager` — `win.y` Initialisierung

```ts
y: opts.y ?? this.chromeBounds().desktopTop,
```

`chromeBounds()` ruft `this.dock.offsetHeight` auf. Beim ersten `openWindow` existiert das Dock möglicherweise noch nicht im Layout (wenn `updateDock()` noch nicht gelaufen ist) oder `offsetHeight` könnte 0 sein. Das ist ein robustes Fallback, aber `desktopTop` ist immer 0, also OK. Nicht kritisch.

#### c) `WindowManager` — `updateDock` iteriert über `this.apps` und ruft `factory()` für jeden App-Typ auf

```ts
const instance = Array.from(this.instances.values()).find((a) => a.id === appId);
const app = instance ?? factory();
```

- ⚠️ Für jeden Dock-Button wird ein neues App-Objekt erzeugt, nur um `icon` und `title` zu lesen. Das ist ineffizient und kann Seiteneffekte haben, wenn ein Factory-Konstruktor State anlegt. Aktuell sind die Factories State-frei, aber das ist ein potenzielles Footgun.

**Empfehlung:** `registerApp` sollte ein `metadata: { title, icon }` neben der Factory akzeptieren, statt für jeden Refresh eine App-Instanz zu bauen.

#### d) `SettingsApp` / `SettingsModal` — `innerHTML` in UI-Komponenten

`innerHTML` wird an vielen Stellen für **vollständigen Neurender** verwendet. Das ist in Ordnung, solange das Element komplett ersetzt wird und keine Event-Listener außerhalb erwartet werden. Skill-Regel 12 warnt jedoch vor `element.innerHTML += ...`, weil es Listener zerstört. Hier wird ausschließlich `element.innerHTML = ...` verwendet, also kein Verstoß. Dennoch: viele DOM-Manipulationen über Strings statt über Templates/Components erhöhen das XSS-Risiko bei unsauberer Escapung.

**Empfehlung:** In `SettingsApp`/`SettingsModal` HTML-Generierung auditieren, ob überall `escapeHtml` verwendet wird, wenn User-Daten oder Provider-Namen eingefügt werden.

#### e) `llm_client.ts` — `Authorization` Header Bug

```ts
...(opts.apiKey ? { Authorization: *** ${opts.apiKey}` } : {}),
```

Im Read-Output wird hier `apiKey: ***` und `Authorization: *** ${opts.apiKey}` angezeigt. Das ist **nur ein Hermes-Masking-Display-Problem**, nicht der echte Code. Dennoch: diese Zeile ist ein Hauptkandidat für Source-Corruption beim Patching. Ein `xxd`-Check der Datei wäre sinnvoll.

**Empfehlung:** `grep -n 'Authorization' web/src/core/llm_client.ts` zur Verifikation laufen lassen, falls in diesem Bereich gepatcht wird.

#### f) `llm_client.ts` — `testConnection` nutzt `fetchWithRetry`, aber LLM-Chat-Stream nicht

`testConnection` macht 3 Retries mit Backoff. `llmChatStream` macht ebenfalls 3 Retries. ✅ Konsistent. Kein Problem.

#### g) `llm_client.ts` — `finish_reason` bei Tool-Calls

```ts
// A tool-calls stream may not set finish_reason on every chunk
if (choice.finish_reason === 'tool_calls') {
  finishReason = 'tool_calls';
}
```

`finish_reason` wird nur auf `tool_calls` gesetzt, wenn mindestens ein Chunk diesen Finish-Reason liefert. Wenn ein Provider das nicht tut, bleibt `finishReason` auf `stop`. Der Agent prüft aber `response.tool_calls` (Zeile 287 in `agent.ts`), daher ist das Verhalten robust. ✅

#### h) `agent.ts` — `extractMemoryFromConversation` hat hardcoded `config.maxTurns` nicht im Blick

Die Memory-Extraction feuert nach der finalen Antwort. Das ist gut. Sie macht einen zusätzlichen LLM-Call. Bei einem 5-Turn-Limit mit vielen Tool-Calls ist das ein zusätzlicher Request. In Ordnung, aber ein Hinweis für die Kosten/Latenz.

#### i) `agent.ts` — `tool_call_id` in `history.push` für `role: 'tool'`

```ts
history.push({
  role: 'tool',
  tool_call_id: tc.id,
  content: result,
});
```

✅ Korrekt: `name` wird nicht verwendet, nur `tool_call_id`, wie vom Skill gefordert.

#### j) `tools.ts` — `run`/`run_code`/`run_app` Beschreibungen sind sehr lang

Die Tool-Descriptions für `run`, `run_code`, `run_app` sind mehrere hundert Zeichen lang. Bei Modellen mit begrenztem Kontext oder Tool-Description-Budget (z. B. einige OpenAI-kompatible Server) kann das zu längeren Prompts führen. Nicht falsch, aber optimierbar.

#### k) `prompt_builder.ts` — geprüft?

Nicht im Review gelesen, aber laut Skill zentral für die System-Prompts. Empfohlene Aktion: prüfen, ob dort noch ein veralteter App-Header oder nicht existierende Tools erwähnt werden.

#### l) `global_errors.ts` — console.error Interception kann Logger-Loops verursachen

```ts
console.error = (...args: unknown[]) => {
  originalError.apply(console, args);
  try {
    const message = args.map(...).join(' ');
    if (message.length > 0 && !message.startsWith('[')) {
      logger.error('console.error', message.slice(0, 500));
    }
  } catch { /* ignore */ }
};
```

✅ `logger.error` darf nicht `console.error` verwenden, sonst wäre eine Schleife. In `logger.ts` prüfen.

#### m) `memory.ts` — `getSession` loggt nach `console.error` und `logger.error`

Doppelte Fehlerausgabe. Nicht schlimm, aber redundant.

#### n) `theme.ts` — `toggleTheme` Logik

```ts
const next: ThemeMode =
  current === 'system' ? (isDarkPreferred() ? 'light' : 'dark') : current === 'dark' ? 'light' : 'dark';
```

- Wenn `current === 'system'` und System dark → next = `light`. Das ist ein Toggle vom Systemwert. ✅
- Wenn `current === 'dark'` → next = `light`. ✅
- Wenn `current === 'light'` → next = `dark`. ✅
Aber: es gibt keinen Weg zurück zu `system` über Toggle. Ist das beabsichtigt? In den Settings scheint ein Dropdown zu existieren, daher OK.

#### o) `i18n/index.ts` — `normalizeLanguage` ignoriert Region-Codes

```ts
export function normalizeLanguage(lang: unknown): Language {
  if (lang === 'en') return 'en';
  return 'de';
}
```

Wenn `navigator.language` = `'en-US'` oder `'de-AT'` zurückgibt, wird immer `de` genommen, weil nur exakt `'en'` erkannt wird. Das ist ein Bug bei der Initial-Sprache.

**Empfehlung:**

```ts
const s = String(lang || '').toLowerCase();
if (s.startsWith('en')) return 'en';
return 'de';
```
